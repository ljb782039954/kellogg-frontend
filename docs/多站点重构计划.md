# 多站点重构计划

本文档基于 `docs/需求文档.md` 和当前项目现状编写，目标是把当前 Kellogg 单站点前端，逐步重构为“通用业务逻辑集中管理 + 站点资源包集中管理 + 构建时切换站点入口”的多站点架构。

## 1. 重构目标

### 1.1 核心目标

1. `src/core` 只承载跨站点通用能力：
   - API 请求封装
   - 请求内去重
   - 汇率/价格工具
   - HTML 消毒
   - 视频安全解析
   - CSP/安全响应头生成
   - 通用 hooks/services
   - 通用 UI 组件，例如 `OptimizedImage`、`TurnstileWidget`、`RichText`

2. `src/site-package/{siteName}` 承载站点专属资源：
   - 站点配置
   - 站点 Layout 壳
   - Header/Footer/Tawk 等品牌组件
   - styles
   - types
   - pages
   - block 组件和 block 映射表
   - 站点专属默认文案、SEO、第三方服务配置
   - 新增站点资源时，除 `pages/` 页面与页面级 Astro wrapper 外，不再新增 `.astro` 组件；站点普通组件优先使用 `.tsx`、`.ts` 或样式资源。当前已有的 `Footer.astro`、`TawkChat.astro`、`SocialLinks.astro`、`ProductCustomFields.astro`、`SiteLayout.astro` 暂作为兼容债务保留。

3. `src/pages` 仅保留 Astro 路由入口：
   - 不直接引用 `src/site-package/kellogg`
   - 不写 Kellogg 业务逻辑
   - 只读取当前站点配置，并委托给当前站点包中的页面实现

4. 站点切换由构建时环境变量决定：
   - 建议使用 `PUBLIC_SITE_NAME`
   - 默认可为 `kellogg`
   - 切换站点时，入口不改业务代码，只改站点名或构建环境

### 1.2 非目标

1. 本轮不要求一次性支持多个线上站点同时运行。
2. 本轮不要求重写现有视觉设计。
3. 本轮不要求改动后端 API 结构，除非前端无法完成站点隔离。
4. 本轮不运行全量构建，构建由人工操作。
5. 测试只覆盖当前批次相关内容，不做全量测试。

## 2. 当前现状判断

当前项目已经有初步目录雏形：

```text
src/
├─ core/
├─ pages/
└─ site-package/
   └─ kellogg/
```

但目前还没有真正完成多站点架构，主要问题如下。

### 2.1 core 仍然反向依赖站点包

`src/core/components/DynamicRenderer.astro` 直接引用了：

```text
../../site-package/kellogg/components/blocks/...
```

`SectionHeader` 这类带有站点视觉语义的 base UI，应归入 Kellogg 站点包，而不是放在 core。

这违反了依赖方向：

```text
正确方向：site-package/{siteName} -> core
错误方向：core -> site-package/{siteName}
```

### 2.2 src/pages 仍然绑定 Kellogg

多个路由入口直接引用：

```text
../site-package/kellogg/layouts/SiteLayout
../site-package/kellogg/components/...
```

例如：

- `src/pages/[...slug].astro`
- `src/pages/products.astro`
- `src/pages/inquiry.astro`
- `src/pages/customer-reviews.astro`
- `src/pages/blog/index.astro`
- `src/pages/blog/[slug].astro`
- `src/pages/product/[id].astro`

这意味着即使新增第二个站点，路由层仍会固定渲染 Kellogg。

### 2.3 站点配置入口缺失

目前缺少类似：

```text
src/site-package/kellogg/config.ts
src/site-package/index.ts
```

也缺少统一的 `getCurrentSite()` 或 `currentSite` 机制。

`SiteService` 仍然硬编码：

```ts
const SUPPORTED_LANGUAGES = ["zh", "en"];
const DEFAULT_LANGUAGE = "en";
```

这些应该改为由站点包配置提供。

### 2.4 API 与资源配置仍散落在 core 中

`src/core/lib/api.ts` 直接读取：

```ts
import.meta.env.PUBLIC_API_BASE_URL
import.meta.env.PUBLIC_API_BASE_URL_LOCAL
import.meta.env.PUBLIC_API_ASSETS
```

并且还存在 Kellogg 域名判断：

```ts
url.includes("kelloggfashion.com")
```

这会让 core 仍然携带站点信息。后续应该改为由站点配置注入。

### 2.5 Layout 还没有拆成 core 骨架与站点壳

当前 `src/site-package/kellogg/layouts/SiteLayout.astro` 同时负责：

- HTML 文档结构
- SEO
- meta
- Header/Footer
- Tawk
- 汇率注入
- 固定 hreflang
- Kellogg 默认标题
- 全局 CSS

需求中的目标是：

```text
core: 通用 HTML 骨架、SEO、lang、script 注入
site-package: 品牌壳、Header/Footer、主题 CSS、第三方服务
```

### 2.6 类型有重复和混用

目前 `src/core/types.ts` 和 `src/site-package/kellogg/types/common.ts` 都定义了 `Language`、`Translation`。

建议后续明确：

- core 只保留最小通用协议。
- 站点包保留站点内容模型。
- 站点包类型可以组合或收窄 core 类型。

### 2.7 测试仍有旧目录假设

部分测试仍在读取旧路径：

```text
src/components/blocks
src/functions/DynamicRenderer.astro
src/lib/hydrationState.ts
src/hooks/useInquiry.ts
```

这些路径已经不符合当前迁移后的目录结构。重构过程中需要同步调整相关测试，否则测试会失去保护意义。

### 3.2 依赖方向

允许：

```text
src/site-package/kellogg -> src/core
src/pages -> src/site-package/index
src/pages -> src/core/services/siteService
```

禁止：

```text
src/core -> src/site-package/**
src/pages -> src/site-package/kellogg/**
```

`src/pages` 只允许引用站点选择入口，例如：

```ts
import { currentSite } from "../site-package";
```

不要直接引用具体站点：

```ts
// 不建议
import Layout from "../site-package/kellogg/layouts/SiteLayout";
```

## 4. 核心接口设计建议

### 4.1 core 类型

`src/core/types.ts` 建议只放跨站点最小协议。

```ts
export type Language = string;

export type Translation = Record<string, string>;

export interface CorePageBlock {
  id: string;
  type: string;
  content: Record<string, unknown>;
  isVisible: boolean;
}

export interface CoreCustomPage {
  id: string;
  path: string;
  title: Translation;
  type?: "fixed-block" | "dynamic-block" | "fixed-layout";
  content?: unknown;
  blocks: CorePageBlock[];
  seo?: {
    title?: Translation;
    description?: Translation;
    keywords?: Translation;
    targetCountry?: string;
  };
}

export interface CoreSiteContent {
  pages: CoreCustomPage[];
  lang: Language;
}
```

不建议 core 长期持有完整的 Kellogg `CompanyInfo`、`HeaderContent`、`FooterContent`、`Product`、`Blog` 细节。若 core 需要使用，只声明它真正依赖的最小字段。

### 4.2 站点配置

建议 `src/site-package/kellogg/config.ts` 提供统一配置。

```ts
  localApiBaseUrl: import.meta.env.PUBLIC_API_BASE_URL_LOCAL,
  assetsBaseUrl: import.meta.env.PUBLIC_API_ASSETS,

  defaultCurrency: "USD",

  turnstile: {
    siteKey: import.meta.env.PUBLIC_TURNSTILE_SITE_KEY,
  },

  tawk: {
    scriptUrl: "https://embed.tawk.to/...",
  },

  seo: {
    defaultTitle: "Kellogg Fashion",
    alternates: [],
  },

  security: {
    csp: {
      scriptSrc: ["https://challenges.cloudflare.com", "https://*.tawk.to"],
      frameSrc: [
        "https://challenges.cloudflare.com",
        "https://www.youtube-nocookie.com",
        "https://player.vimeo.com",
        "https://www.facebook.com",
        "https://www.tiktok.com",
        "https://*.tawk.to",
      ],

```

实际落地时可以比上面更简单，不必一次塞满所有字段。关键是先把“站点级常量”集中起来。

### 4.3 站点注册入口

建议新增 `src/site-package/index.ts`。

```ts
import { kelloggSiteConfig } from "./kellogg/config";

const siteConfigs = {
  kellogg: kelloggSiteConfig,
} as const;

export type SiteName = keyof typeof siteConfigs;

export function getCurrentSiteName(): SiteName {
  const siteName = import.meta.env.PUBLIC_SITE_NAME || "kellogg";

  if (siteName in siteConfigs) {
    return siteName as SiteName;
  }

  throw new Error(`[site-package] Unknown site: ${siteName}`);
}

export const currentSite = siteConfigs[getCurrentSiteName()];
```

注意：Astro/Vite 对动态 import 和静态分析比较敏感。站点注册表建议显式 import 所有可选站点配置，不建议一开始就做完全动态路径：

```ts
// 不建议作为第一阶段方案
await import(`./${siteName}/config`);
```

### 4.4 API 客户端

`src/core/lib/api.ts` 建议从单例改为可注入。

```ts
export interface ApiClientConfig {
  apiBaseUrl: string;
  localApiBaseUrl?: string;
  assetsBaseUrl: string;
  siteHostnames?: string[];
}

export function createApiClient(config: ApiClientConfig) {
  const apiBase = resolveApiBase(config);

  return {
    getProducts,
    getProduct,
    getCategories,
    getConfig,
    getPageById,
    getBlogs,
    getBlog,
    getReviews,
    submitInquiry,
    resolveMediaUrl,
    getOptimizedImageUrl,
  };
}
```

这样 `SiteService` 或站点配置可以提供当前站点的 api。

短期兼容方案可以保留：

```ts
export const api = createApiClient(currentSite.api);
```

但要注意不要让 core 直接 import `currentSite`，否则又会形成 core -> site-package。更推荐由 `SiteService` 创建并传递。

### 4.5 SiteService

`SiteService` 的职责建议调整为：

1. 获取当前站点配置。
2. 基于站点配置解析语言。
3. 基于站点配置创建 API client。
4. 请求并缓存当前请求内的站点数据。

伪代码：

```ts
export class SiteService {
  static getConfig() {
    return currentSite;
  }

  static getApi() {
    return createApiClient(currentSite.api);
  }

  static async getSiteData(Astro?: SiteRequestContext) {
    if (Astro?.locals) {
      Astro.locals.siteDataPromise ||= this.loadSiteData(Astro);
      return Astro.locals.siteDataPromise;
    }

    return this.loadSiteData(Astro);
  }

  private static resolveLanguage(requested?: string) {
    const { languages, defaultLanguage } = currentSite;
    return requested && languages.includes(requested)
      ? requested
      : defaultLanguage;
  }
}
```

这里有一个边界点：如果 `SiteService` 在 `src/core/services` 中，但它 import `currentSite`，就会变成 core 依赖站点包。可以有两种处理方式：

1. 接受 `SiteService` 作为“应用层服务”，从 `core` 挪到 `src/site-package` 或 `src/app`。
2. 保持 `SiteService` 在 core，但不直接 import 当前站点，由 `src/pages` 传入站点配置。

考虑当前项目迁移成本，建议第一阶段采用折中方案：

```text
src/core/services/siteData.ts        纯通用数据加载工具
src/site-package/index.ts            当前站点配置入口
src/services/siteService.ts          应用层服务，连接 currentSite 和 core
```

如果暂时不想新增 `src/services`，也可以先保留 `src/core/services/siteService.ts`，但要在后续批次把它移出 core 或拆分，避免长期违反依赖方向。

## 5. 分阶段实施计划

### 阶段 0：建立基线与边界清单

目标：确认当前迁移状态，建立后续不跑偏的检查点。

任务：

1. 记录当前硬编码清单：
   - `site-package/kellogg`
   - `kelloggfashion.com`
   - `Kellogg Fashion`
   - `["zh", "en"]`
   - `PUBLIC_API_BASE_URL`
   - `PUBLIC_API_ASSETS`
   - Tawk 固定 script URL
   - Turnstile 固定 production key

2. 记录当前跨层 import：
   - core -> site-package
   - pages -> site-package/kellogg
   - tests -> 旧目录

3. 暂不改业务行为，只补充轻量架构测试或脚本。

建议新增测试：

```text
tests/multisite-architecture.test.mjs
```

检查：

- `src/core` 不出现 `site-package/`
- `src/pages` 不出现 `site-package/kellogg`
- `src/site-package/index.ts` 存在
- 当前站点配置存在
- 新增站点包 `.astro` 文件不得出现在 `pages/` 外，已有兼容债务使用白名单保护

验收：

1. 能列出所有硬编码点。
2. 有最小架构测试保护后续迁移。
3. 不要求页面行为变化。

### 阶段 1：新增站点配置与注册表

目标：先让项目有“当前站点”的统一入口。

任务：

1. 新增：

```text
src/site-package/kellogg/config.ts
src/site-package/index.ts
```

2. 将以下配置先集中到 Kellogg config：

- 站点名
- 默认标题
- 支持语言
- 默认语言
- site URL
- API base URL
- assets base URL
- Turnstile site key
- Tawk script URL
- CSP 第三方域名
- hreflang alternates
- 默认货币

3. 暂时不要求所有使用点立即切换，但新代码必须从配置读取。

验收：

1. `PUBLIC_SITE_NAME=kellogg` 能解析到 Kellogg config。
2. 未知 `PUBLIC_SITE_NAME` 抛出明确错误。
3. 后续新增站点只需要新增一个 `site-package/{site}/config.ts` 并注册。

### 阶段 2：语言与翻译工具解耦

目标：彻底去掉通用层对 `zh/en` 的默认假设。

任务：

1. 保留 core 中的通用类型：

```ts
export type Language = string;
export type Translation = Record<string, string>;
```

2. 修改 `createTranslate()` 和 `t()`：

- fallback 语言从站点配置传入
- 默认兜底为：
  1. 当前语言
  2. 显式 fallback
  3. 站点 fallback languages
  4. translation 对象第一个可用值
  5. 空字符串

3. 修改 core 组件，不再引用 Kellogg 类型。

重点文件：

- `src/core/lib/i18n.ts`
- `src/core/lib/common.ts`
- `src/core/components/TurnstileWidget.tsx`

验收：

1. `src/core` 不再从 `src/site-package/kellogg/types` 引入类型。
2. 新增语言 key 不需要改 core 类型。
3. 站点可以声明自己的语言列表与默认语言。

### 阶段 3：API 与站点资源注入

目标：让 API endpoint、assets base、优化图片域名判断全部由站点配置控制。

任务：

1. 将 `api` 单例拆成 `createApiClient(config)`。

2. 将以下配置从 core 移出：

- `PUBLIC_API_BASE_URL`
- `PUBLIC_API_BASE_URL_LOCAL`
- `PUBLIC_API_ASSETS`
- `kelloggfashion.com`

3. `resolveMediaUrl()` 使用 `assetsBaseUrl`。

4. `getOptimizedImageUrl()` 使用站点配置判断是否可走站点图片优化规则，例如：

```ts
siteAssetHosts: ["kelloggfashion.com", "cdn.kelloggfashion.com"]
```

5. 商品筛选、博客、评价、询盘提交继续通过统一 API client 发起。

验收：

1. core API 不包含 Kellogg 域名。
2. 更换站点配置后，API 和资源域名能随配置切换。
3. 客户端组件仍能取消过期请求。

### 阶段 4：拆分 Layout

目标：把通用 HTML 骨架和站点品牌壳拆开。

任务：

1. 新增：

```text
src/core/layouts/CoreLayout.astro
```

职责：

- `<!doctype html>`
- `<html lang>`
- charset
- viewport
- referrer
- generator
- title
- description
- keywords
- alternate links
- exchange rates script 注入
- 其他通用 head script 注入
- body slot

2. 修改：

```text
src/site-package/kellogg/layouts/SiteLayout.astro
```

职责：

- 引入 Kellogg global.css
- 获取站点数据
- 传 SEO 数据给 CoreLayout
- 渲染 Header
- 渲染 TawkChat
- 渲染页面 slot
- 渲染站点级 InquirySection
- 渲染 Footer
- 初始化价格追踪

3. 固定 hreflang 从代码中移到站点配置。

4. 默认标题 `Kellogg Fashion` 从代码中移到站点配置。

验收：

1. core Layout 不引用 Kellogg 组件。
2. Kellogg SiteLayout 可以自由组合 core Layout。
3. 站点级 SEO alternate links 来自配置。

### 阶段 5：重构 DynamicRenderer 与 block map

目标：core 只负责通用渲染流程，站点包负责 block 类型与组件映射。

任务：

1. 新增：

```text
src/site-package/kellogg/blockMap.ts
```

2. block map 中集中声明：

- block type
- component
- hydration 策略
- 是否需要全局数据
- 数据预取函数或数据需求声明

示意：

```ts
export const blockMap = {
  imageBanner: {
    component: ImageBanner,
    hydration: "none",
  },
  carousel: {
    component: Carousel,
    hydration: "load",
  },
  productGrid: {
    component: ProductGrid,
    hydration: "load",
    loadData: async ({ block, api, requestMemo }) => {
      // products/categories
    },
  },
};
```

3. 修改 `src/core/components/DynamicRenderer.astro`：

- 删除所有 Kellogg block import
- 通过 props 接收 `blockMap`
- 根据 blockMap 做数据注入和渲染
- 未注册 block 在开发环境显示提示，生产环境忽略或记录日志

4. 注意 Astro 的水合限制：

Astro 的 `client:load`、`client:visible` 是编译期语法，不能简单地把组件存进对象后动态加 `client:*`。如果动态 map 难以直接承载水合策略，建议让站点包提供一层 Astro renderer wrapper：

```text
src/site-package/kellogg/pages/BlockRenderer.astro
```

由这个 wrapper 显式写：

## 7. 关键风险与处理建议

### 7.1 Astro 动态组件与 hydration 风险

风险：

`client:load`、`client:visible` 等 Astro hydration 指令不是普通运行时 props，不能保证通过动态 map 自动生效。

建议：

对需要水合的 block，优先使用站点包 Astro wrapper 明确写出水合指令。不要在第一版里追求过度抽象。

### 7.2 core 过度膨胀风险

风险：

如果把 `Product`、`Blog`、`CompanyInfo`、`HeaderContent` 全部放进 core，未来第二个站点会被 Kellogg 的业务模型限制。

建议：

core 只保留最小通用协议。站点包可以定义更完整的类型，并在调用 core 时收窄到 core 所需字段。

### 7.3 站点配置过度复杂风险

风险：

一开始把所有东西抽象成插件式配置，会导致迁移成本过高。

建议：

第一阶段只集中真实已经散落的常量。等第二个站点出现后，再根据真实差异抽象。

### 7.4 路由委托过早动态化风险

风险：

Astro 对动态 import、动态组件渲染、静态路径生成有编译期限制。过早做完全动态路由可能影响预渲染。

建议：

先用显式注册表，保证 Vite/Astro 可静态分析。必要时每个 `src/pages` 仍保留物理文件，但内部只做站点分发。

### 7.5 sitemap 与页面能力不一致

风险：

如果 sitemap 继续枚举所有 CMS 页面和固定页面，而当前站点未实现某个页面，会输出无效 URL。

建议：

sitemap 应基于：

- 当前站点注册的页面能力
- CMS pages
- 商品/博客能力开关
- 页面是否 active/visible

### 7.6 CSP 配置遗漏

风险：

拆 CSP 后容易漏掉 Tawk、Turnstile、视频 iframe 域名，导致线上第三方组件失效。

建议：

先写 CSP builder 测试，验证 Kellogg 当前所需域名仍存在。

### 7.7 请求缓存边界

风险：

为了多站点切换，如果误用模块级缓存，Cloudflare isolate 可能跨请求共享旧站点数据。

建议：

继续保持请求内缓存放在 `Astro.locals` 或请求内 `Map`，不要引入模块级 API 响应缓存。

## 10. 建议优先级

如果只能先做一小批，建议优先做：

1. 新增 `site-package/index.ts` 和 `kellogg/config.ts`。
2. 修复 core -> Kellogg 类型依赖，并将站点视觉组件保留在 `site-package/kellogg/components/base`。
