# 多站点重构需求文档

将当前单一站点（Kellogg）重构为可切换站点的多站点架构。

## 1. 目录与职责划分

1. `src/core` 承载跨站点共享的底层能力：API 请求、汇率/价格、视频白名单、HTML 消毒、请求去重、通用 hooks/services、通用 UI 组件（`OptimizedImage`、`TurnstileWidget`、`TawkChat` 等）。
2. `src/site-package/{siteName}` 承载站点专属资源：components、语言、styles、types、pages，以及 Header/Footer 品牌壳。当前站点命名为 `kellogg`。
3. `src/pages` 保留为 Astro 路由入口，但每个文件仅负责读取当前站点配置并委托给 `src/site-package/{siteName}/pages` 中的同名页面；站点包缺失的页面返回 404，sitemap 同步过滤。

## 2. 类型与语言解耦

1. `Language` 与 `Translation` 不能写死为 `zh/en`。而是应该由站点决定
2. `Layout.astro` 拆分为 core 层通用骨架（SEO、HTML lang、CSP 数据、script 注入）和站点包品牌壳（Header/Footer 组件实例、主题 CSS、Tawk 配置）。core 通过 Props 接收站点壳组件，避免反向依赖站点包。
3. `utils/common.ts` 中的 `t()` 删除对站点类型的耦合，迁为 core 工具，支持按任意语言 key 兜底读取；站点包负责提供语言列表与默认语言。
4. 站点内容类型（`CompanyInfo`、`HeaderContent`、`FooterContent`、`CustomPage` 等）保留在站点包 `types` 中；core 仅声明它需要的最小接口，站点包通过类型收窄或组合满足。

## 3. 站点切换机制

8. 当前站点由构建时环境变量（如 `PUBLIC_SITE_NAME`）决定，`SiteService` 读取对应 `site-package` 的配置入口，其他代码不再硬编码 `kellogg`。
9. 各站点 API endpoint、assets base、站点 URL、Tawk/Turnstile key、货币默认值等全部通过站点配置注入，core 不保留站点级常量。

## 4. 积木与页面渲染

10. `DynamicRenderer.astro` 迁入 core，负责按 block 类型分发渲染；站点包提供 block 到组件的映射表（map），core 按 map 加载组件。新增 block 类型时站点包扩展 map，core 无需改动。
11. 页面级组件（`InquiryView`、商品详情、博客等）保留在站点包，可自由组合 core 通用组件与站点样式。

## 5. 边界约束

12. 依赖方向严格为：站点包 → core。core、通用组件、通用 hooks 禁止引用任何 `src/site-package/**` 路径。
13. `middleware.ts`、安全响应头、CSP 配置保留在 core；涉及第三方域名的白名单由站点配置补充，避免在 core 写死 Tawk/Turnstile 域名。

## 6. 注意
14. 站点资源中不要有.astro组件，除了pages/页面以外，这是为了增加兼容性，已有的先暂时不要修改。
